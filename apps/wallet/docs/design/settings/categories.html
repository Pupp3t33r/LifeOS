Categories — management (Settings → Categories)

The dedicated surface where you shape the category list: search, rename and recolour your own, recolour the built-ins, archive and restore. Colour is a category’s identity everywhere else in the app — Activity rows, split bars, budgets — so recolouring is a pick from the 12-slot Calm palette, a finite named set, not a colour wheel. Palette board + light/dark renderings live in home/categories.html; this file specifies the management screen itself.

Rename to free a name — e.g. “Books” → “Books_old” — before reintroducing it fresh and narrower.

Search

  • A field pinned at the top filters Yours + System + Archived by name (case-insensitive substring), mirroring the search on settings.html. Empty groups collapse out of the results.
  • Archived is always in scope — that’s the point: a retired category is easiest to find by typing its name. Matches surface even while the Archived section is otherwise collapsed; open a result to rename or restore it.

Archive ↔ Unarchive

  • Archive (clay, from a live category’s opened row) retires it: gone from the picker and the active groups, kept under Archived. Historical lines and budgets still resolve by id — nothing is deleted (soft-archive, ADR-0024).
  • Archived rows stay editable. Open one to recolour, Rename, or Unarchive (sage, the inverse of Archive). Only the Archive action drops — it’s already archived — and system categories can’t be archived at all.
  • Archived defaults collapsed (a count in the eyebrow); tap to expand, or let search reveal a match. The row stays dimmed to read as retired even while its editor is open.

Naming, identity & uniqueness (decided)

  • Keyed by Guid Id — client-assigned for user categories (ADR-0003), fixed constants for the three system ones. The id is what a spending line references; the name is display only. Backend has no Schema.For<Category>() yet, so identity would default to Id with no name constraint — this rule adds it.
  • Names are unique per owner, case-insensitive and trim-normalised (“Food” = “food ”). The check spans every user category — active and archived — plus the three system names, so there’s no user “Books” shadowing the system one and no two “Food”s.
  • Archived names stay reserved so Unarchive can never collide. To reintroduce a name, first open the archived holder and rename it (“Books” → “Books_old”) — that frees “Books” for a fresh, narrower category (e.g. splitting Books from Comics).
  • Enforced server-side — a create/rename validator plus a partial unique index (owner_id, lower(name)) where owner_id is not null (covers active + archived; system rows aren’t in the table) — and mirrored client-side for an inline error before the outbox round-trips. The server stays source of truth since offline creates can race.

Reached from a single navigation row on settings.html (“Categories · Manage your categories”) → route /settings/categories, a full screen above the shell with a back affordance (same level as /settings). It’s a CRUD list with its own affordances, not an inline control, and ADR-0003 names “Settings → Categories” as its home.

Signature — the inline named palette

  • Tapping a category’s swatch (enlarged from the app’s 11 px dot to a 30 px identity tile) slides the 12 Calm slots open in place; the selected slot is ringed and named (“Clay”). Recolouring is a pick from a finite, curated set — the same twelve colours that identify categories in Activity and budgets.
  • Evolves the earlier Settings→Categories sketch in home/categories.html, which used a floating popover colour picker. The inline strip reads better on a phone and puts Rename & Archive in the same opened context.

Decisions baked in (ADR-0024 / ADR-0003)

  • Yours = user categories, full lifecycle: create (ghost row), rename, recolour, archive.
  • System = Books / Board Games / Video Games. Swatch stays (recolourable, ADR-0003); the row carries a system lock chip and drops Rename & Archive (names locked, ADR-0024).
  • Colour never travels to the Money domain — it’s a device-local display override, deterministic default from CategoryId. New categories get the next-free slot, editable.
  • Cut from the earlier sketch: per-category entry counts and the “how they read on the canvas” preview strip — spend is a lens in Budgets/Activity, so this screen stays purely about identity (name + colour + lifecycle).

Specified in — ADRs

  • Money ADR-0033 — the write path: client-assigned id + Marten registration, the name-uniqueness invariant (validator + partial unique index), reversible archive/unarchive, archived-but-editable, and the ?includeArchived management read (amends ADR-0024).
  • Wallet ADR-0008 — this screen + the device-local colour-override store (realizes ADR-0003): placement at /settings/categories, search-incl-archived, the inline named palette, CRUD via the offline outbox, client-mirrored uniqueness.

Not buildable yet — needs backend + client work

  • Money API: only GET /categories exists today. Create / rename / archive / unarchive per ADR-0033 are unbuilt.
  • Wallet client: no device-local colour-override store yet (ADR-0003/0008). Recolour + search are the pieces buildable first (search filters whatever the overlay already returns); create/rename/archive/unarchive wait on the API.
