# ADR-0003: Category colour system — a curated 12-colour Calm palette, client-assigned, light/dark

## Status

Accepted

Date: 2026-06-29

**Relates to:** [Money ADR-0024](../../../../services/money/docs/adr/0024-category-model.md) (the managed category list — system + user categories), [Money ADR-0019](../../../../services/money/docs/adr/0019-universal-line-items.md) (per-line `CategoryId` — the thing being coloured), [Wallet ADR-0001](./0001-app-localization.md) (the client-only, device-local preference precedent the colour overrides follow), [Wallet ADR-0002](./0002-navigation-and-information-architecture.md) (the row proportion bars, per-line dots, and budget bars that consume these colours).

## Context

The Home cockpit (Wallet ADR-0002) renders categories visually in three places: the **proportion bar** on a container row (a multi-category entry shows its split), the **per-line dots** in an expanded row, and the **budget bars** in the on-track expand. For any of that, a category needs a colour.

Colours are seductive and easy to get wrong. A free colour wheel, or one saturated hue per category, produces a garish result that fights the restrained "Calm" theme (bone / sage / clay / neutrals) — and it must stay legible in **both** light and dark mode and remain distinguishable. Categories are also open-ended: the user can create any number (Money ADR-0024), and there are immutable **system** categories from other domains (Books, Board Games, Video Games).

Forces at play:

- **The Calm theme is the brand**, already shipped (onboarding, Keycloak login). Category colours must extend it, not introduce a competing palette.
- **Light and dark are both first-class.** Every theme token in `calm_tokens.dart` already carries a light and a dark value; category colours must too (a single hex will muddy on one mode or the other).
- **Colour changes nothing the server computes.** Per the Wallet architecture rule (root + `apps/wallet/AGENTS.md`) and the precedent in Wallet ADR-0001, presentation-only preferences are client-side, not Money `UserPreferences`. A category's colour is presentation.
- **System category names are domain constants** (Money ADR-0024) — immutable. But a *colour* is not a name; recolouring a system category is a personal display choice with no domain effect.
- **Defaults should be stable across devices** even without a sync channel — a category shouldn't be sage on the phone and plum on the desktop by accident.

## Decision

### A curated, fixed 12-colour palette

Category colours are chosen from a **fixed set of 12 named slots**, not a free picker:

> Sage · Teal · Denim · Indigo · Plum · Rose · Rust · Clay · Ochre · Olive · Stone · Slate

Each slot is a **muted mid-tone in the Calm family** with a **separate light and dark rendering** (dark lifts luminance so the bars read on the dark bone). The palette is added to the Calm token set (alongside the existing `calm_tokens.dart` / `design/themes/calm/tokens.json` values, regenerated by Style Dictionary when it lands). Sage and Clay coincide with the brand primary/accent, so categories feel of-a-piece with the app. Twelve is enough to keep categories distinct and few enough to stay calm.

### Each category maps to one slot; defaults are deterministic

- A category's colour is a single palette slot. The **default** slot is derived **deterministically from the `CategoryId`** (a stable hash into the 12 slots), so a category gets the same colour on every device with no sync, and distinct categories spread across the palette.
- The user can **reassign** any category's colour in **Settings → Categories** (the 12-swatch picker). This includes the **system** categories: their *names* stay locked (Money ADR-0024), but **Recolour** is allowed.
- Reuse of a colour across categories is **allowed but discouraged** (the picker notes it) — distinct colours keep the split bars legible.

### Colour is a client/display concern, stored device-local

- A category's colour is **never** stored on Money's `Category` (Money ADR-0024) — Money carries no colour field. It is presentation.
- User **overrides** are stored **device-local**, keyed by `CategoryId`, in the same client-preference layer as the locale (Wallet ADR-0001) and app-lock (Money ADR-0014) — i.e. `flutter_secure_storage` / local prefs via a small store, not the server.
- **Cross-device sync of overrides is deferred.** Defaults are already cross-device-stable (deterministic from `CategoryId`); only manual overrides are per-device until a client-preferences sync exists. Acceptable for solo v1.

### What consumes the colour

The resolved colour (override if present, else the deterministic default) feeds: row **proportion bars** and **per-line dots** (Wallet ADR-0002), **budget bars** (Money ADR-0025), and any future surface where a category renders. A single resolver (`CategoryColor.of(categoryId)`) maps `CategoryId → slot → mode-aware Color`.

## Consequences

Positive:

- Twelve curated Calm tones avoid the garish, palette-exploding failure mode of a free colour wheel, and read in both modes.
- Defaults are stable across devices with zero sync, because they derive from `CategoryId`.
- System categories integrate (recolourable) without violating their locked names or adding colour to the domain.
- Colour sits at the right layer — a client display preference — adding nothing to Money's surface.

Negative:

- A fixed 12-slot palette caps simultaneous *distinct* category colours; users with many categories will reuse colours (discouraged but allowed). Accepted: the expanded-row and budgets views carry the unambiguous per-line truth.
- Manual overrides don't sync across devices until a client-prefs sync exists; a recolour on the phone won't appear on the desktop. Mitigated by stable defaults.

Neutral:

- Deterministic default assignment can occasionally place two nearby categories on adjacent hues; the user can recolour. A collision-avoiding allocator is a possible later refinement.
- The palette hexes will be generated from `tokens.json` once Style Dictionary lands, like the rest of the Calm tokens.

## Alternatives Considered

1. **Free colour picker (full wheel / hex).** Rejected: the direct path to a garish, off-brand, mode-fragile result; the whole point is curation.
2. **One saturated hue per category.** Rejected: vibrates against Calm, especially in dark mode (demonstrated in design exploration).
3. **Store the colour on Money's `Category`.** Rejected: colour changes nothing the server computes; it violates the server-owns-only-server-affecting-config rule and forces one colour across all surfaces/devices regardless of theme.
4. **Random default colour per category.** Rejected: not stable across devices and prone to collisions; a deterministic `CategoryId`-derived default is stable and spread.
5. **No colours — text/dot only.** Rejected here (kept as the fallback in ADR-0002's row exploration): the proportion bar was the preferred treatment, which requires a colour system; this ADR makes it safe.
