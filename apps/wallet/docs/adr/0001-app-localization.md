# ADR-0001: In-app localization — `flutter_localizations` + `intl` (ARB), English + Russian, client-only locale

## Status

Accepted

Date: 2026-06-29

**Relates to:** [Keycloak ADR-0001](../../../../aspire/LifeOS.AppHost/keycloak/docs/adr/0001-login-page-internationalization.md) (the login pages are localized EN + RU; this ADR brings the *app* in line) and [Money ADR-0013](../../../../services/money/docs/adr/0013-user-preferences-and-configurable-month.md) (`UserPreferences`; this ADR explains why the language choice is **not** one of them).

## Context

The Wallet's primary user is in Belarus and prefers Russian ([Money ADR-0015](../../../../services/money/docs/adr/0015-fx-rate-sourcing-and-client-cache.md)). The Keycloak login surface was just localized to English + Russian (Keycloak ADR-0001), but the app itself shipped with hard-coded English strings throughout (onboarding, etc.), so a Russian-speaking user signs in in Russian and then hits an English app — an inconsistent seam.

Forces at play:

- **Flutter has one canonical i18n path.** `flutter_localizations` (the SDK localization delegates) + `intl`, with `gen-l10n` compiling ARB resource files into a typed `AppLocalizations` class wired through `MaterialApp`. It is the documented default, needs no third-party packages, and gives compile-time-checked string access. There is no reason to invent an alternative.
- **Language is presentation, not server state.** The Wallet architecture rule (root `AGENTS.md`, `apps/wallet/AGENTS.md`) is that settings which change **what the server computes** belong on the server (Money `UserPreferences`, ADR-0013 — `MonthStartDay`, `DisplayCurrency`). The UI language changes nothing the server computes; it is purely how the client renders. So it does not belong in `UserPreferences`.
- **There is an established home for client-only, per-device preferences.** The biometric app-lock (ADR-0014) is stored device-locally in `flutter_secure_storage` via `AppLockStore`, explicitly *not* on the server, because it is per-device. Language is the same shape: you may want your phone in Russian and your desktop in English.
- **There is no standalone settings screen yet.** Onboarding is the first (and currently only) settings-collection surface. The language control has to live somewhere reachable from the very first screen, and the flow that hosts it should itself be readable in the chosen language.
- **Generated code is not committed.** A prior decision (`chore(flutter): remove generated files from git`) keeps generated Dart (e.g. drift `.g.dart`) out of version control; only sources are tracked.

## Decision

Adopt Flutter's standard localization stack and ship **English + Russian** in the app, matching the login pages.

### Libraries and tooling

- Add `flutter_localizations` (SDK) and `intl` to `pubspec.yaml`; set `flutter: generate: true`.
- Configure `gen-l10n` via `l10n.yaml`: ARB sources in `lib/l10n/` (`app_en.arb` is the template, `app_ru.arb` the translation), generating a non-nullable `AppLocalizations` (`AppLocalizations.of(context)`).
- **Track the ARB sources; gitignore the generated `lib/l10n/app_localizations*.dart`** (regenerated by `flutter pub get` / `flutter gen-l10n`), consistent with the no-generated-code-in-git policy.
- Wire `MaterialApp.router` with `localizationsDelegates: AppLocalizations.localizationsDelegates`, `supportedLocales: AppLocalizations.supportedLocales`, and `locale:` driven by the locale controller below.

### The locale choice is client-only and device-local

- The chosen language is stored device-locally in `flutter_secure_storage` via a `LocaleStore`, mirroring `AppLockStore`. It is **never** a Money `UserPreferences` field.
- A Riverpod `LocaleController` (`Notifier<Locale?>`) exposes the current locale; `null` means "follow the device/system locale" (Flutter resolves it against the supported set). It restores the saved choice asynchronously on first build — the first frame is the system locale, then it snaps to the saved language.
- Default behavior with no saved choice: follow the system locale, falling back to English when the system locale is unsupported (Flutter's default resolution against `supportedLocales`).

### How the user chooses a language

- A language switcher lives in the **onboarding AppBar**, always visible, mirroring the Keycloak login-page switcher for cross-surface consistency. Selecting a language applies it **live** (the whole flow re-renders immediately) and persists it.
- It is deliberately **not** a gated onboarding step: language is a client preference, not part of the server-config submit (account → display currency → month start day), and it must be changeable *before* any step so the entire flow is readable in the user's language. Folding it into the stepper would couple a presentation toggle to the server-write flow and disturb the step/progress machinery.

### Translation scope (incremental)

- This ADR localizes the **onboarding/settings surface** (every user-facing string on it) in EN + RU, plus the language switcher. This is the surface that hosts the language control and the app's current settings flow.
- The rest of the app (month overview and future feature screens) is localized **incrementally** as each surface is touched — new user-facing strings go through ARB from now on rather than being hard-coded. A full sweep is not a precondition; English remains the fallback for any not-yet-translated string.

## Consequences

Positive:

- The login → app experience is consistent: a user who signs in in Russian stays in Russian.
- Standard, third-party-free stack with compile-time-checked strings; adding a locale later is dropping in `app_<tag>.arb` and (if needed) extending the switcher.
- Language is modeled at the right layer — a client-only, per-device preference next to the app-lock — so it adds nothing to Money's surface and can differ per device.
- Live switching with no restart; the choice survives relaunch.

Negative:

- New user-facing strings must be authored in ARB (a small, ongoing discipline) instead of inline literals; a forgotten string silently falls back to English.
- ARB translations must be kept in sync as English copy changes — the same manual coupling noted for the three Keycloak custom strings.
- A brief first-frame flash from system → saved locale on cold start, because the choice is read from async secure storage. Accepted; gating the whole app on that read is not worth removing a one-frame flash.
- Locale-specific text shaping that ARB does not model (e.g. the day-of-month ordinal — English `25th` vs. Russian `25`) needs small per-locale handling in the widget. Done for the day picker; future such cases are handled case by case.

Neutral:

- Generated `AppLocalizations` is not committed; a fresh checkout must run `flutter pub get` (or `flutter gen-l10n`) before analysis/build resolves the import — the same as drift's generated code.
- Russian was chosen to match the login locales; Belarusian or any other locale is a drop-in later (Keycloak ADR-0001 made the same call for the login pages).

## Alternatives Considered

1. **Store the language in Money `UserPreferences` (ADR-0013).** Rejected: language changes nothing the server computes, so it violates the "server owns only server-affecting config" rule, and it would force one language across all of a user's devices. Client-only, device-local is the correct shape (the app-lock precedent).
2. **A third-party i18n package (e.g. `easy_localization`, `slang`).** Rejected: `flutter_localizations` + `intl` + `gen-l10n` is the first-party path, already integrates with `MaterialApp`/`Localizations`, gives typed accessors, and avoids a dependency for a problem the SDK already solves.
3. **A dedicated onboarding step for language.** Rejected: it must be changeable before the first step (so the whole flow reads in the chosen language) and it is not part of the server-config submit. An always-visible AppBar switcher (matching Keycloak) fits both facts; a step does not.
4. **Localize the entire app in one pass before merging.** Rejected as an unnecessary gate: incremental per-surface localization with an English fallback ships value now (the login-adjacent onboarding surface) without blocking on a full string sweep. The standard fallback behavior makes partial coverage safe.
5. **Commit the generated `AppLocalizations`.** Rejected: contradicts the established no-generated-code-in-git policy; the ARB sources are the tracked artifact and generation is deterministic.
