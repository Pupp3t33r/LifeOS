# ADR-0002: FluentUI Blazor + a board-games-specific theme

## Status

Accepted

Date: 2026-07-07

**Relates to:** [Root AGENTS.md](../../../../AGENTS.md) §6 "Design System" + [`design/README.md`](../../../../design/README.md) (shared theme registry).

## Context

The board-games domain deserves its own visual identity, distinct from Wallet's **Calm** theme (which is tied to the personal-finance planner feel). The shared `design/` registry is built for exactly this — per-theme vocabularies, no shared semantic contract across themes (Calm names its tokens `sage`/`clay`/`bone`; another theme names its own).

Separately, Table needs a Blazor component library. Options considered: MudBlazor (Material-styled), Radzen (its own design language), FluentUI Blazor (Microsoft, neutral, design-token-driven), or hand-rolled components on plain Bootstrap/Tailwind. The choice interacts with theming: a heavy opinionated component lib fights a custom theme; a token-driven one conforms to it.

Forces at play:

- Calm is a custom hand-crafted theme; board games should have the same treatment, not a stock Fluent look.
- FluentUI Blazor is built around design tokens (CSS variables) — theming is a first-class concern, not a fight against hardcoded styles.
- FluentUI Blazor runs in both Blazor Web App and the MAUI `BlazorWebView`, so one component set covers both hosts.
- The shared `design/` registry's rule: a new theme = a new folder under `design/themes/<name>/` with `tokens.json` (authoritative) + bindings. Style Dictionary is still deferred (the trigger is one theme needing >1 generated binding, not 2 themes each with 1).

## Decision

Table uses **FluentUI Blazor** (`Microsoft.FluentUI.AspNetCore.Components`) as its component library, themed by a **new board-games-specific theme** in the shared `design/` registry (name TBD). The theme is distinct from Calm — its own vocabulary, its own token names, its own folder under `design/themes/<name>/`.

- **Theme source of truth:** `design/themes/<name>/tokens.json` (platform-neutral), edited first.
- **CSS binding:** `design/themes/<name>/bindings/tokens.css` — drives FluentUI's design tokens and the web/Blazor CSS. This is the one live binding.
- **MAUI-native surfaces** (status bar, etc.): a small native-color partial maps the theme's key colors to `Microsoft.Maui.Graphics` values. Hand-authored, no generator (Style Dictionary still deferred).
- **No hardcoded color literals** in the app — components reference the theme tokens, same rule as Wallet references Calm.

The theme name is deferred — when it lands, it slots into `design/themes/<name>/` and `design/README.md` is updated to note a second theme exists.

## Consequences

Positive:

- A distinct visual identity for board games, parallel to Calm for finance.
- FluentUI's token-driven theming means the custom theme is a CSS-variable override, not a fork of component internals.
- One component set across both hosts (Web + MAUI `BlazorWebView`).
- Stays within the shared-registry discipline (`design/` owns tokens; consumers reference them).

Negative:

- A second theme is more authoring surface (tokens.json + CSS binding + the MAUI partial). Justified by wanting a distinct identity.
- FluentUI's default look is "Fluent" — making it truly feel like a bespoke board-games theme takes token-tuning effort. Acceptable; that's the theming work.
- Style Dictionary is still deferred — bindings are hand-synced (edit `tokens.json`, mirror into each binding). Fine while each theme has one binding.

Neutral:

- The MAUI-native color partial is the one place FluentUI's CSS tokens don't directly reach (native chrome). Small, hand-maintained.

## Alternatives Considered

1. **MudBlazor.** Rejected: Material-styled and opinionated; theming it away from Material toward a bespoke board-games feel fights the library. FluentUI's token model is more accommodating.
2. **Radzen.** Rejected: rich components but its own design language; heavier and less neutral than FluentUI for a custom-theme project.
3. **Hand-rolled components on Bootstrap/Tailwind.** Rejected for Phase 1: more upfront work to build tables, dialogs, inputs, etc. that FluentUI gives for free. Revisit if FluentUI proves constraining.
4. **Reuse Calm for board games.** Rejected: the team wants board games to have its own identity, and Calm's vocabulary (`sage`/`clay`/`bone`) is finance-flavored.
