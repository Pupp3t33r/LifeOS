# ADR-0005: ViewModels via CommunityToolkit.Mvvm — shared across MAUI + Blazor hosts

## Status

Accepted

Date: 2026-07-07

**Relates to:** [ADR-0001](./0001-blazor-hybrid-stack.md) (the RCL + two-host shape that makes ViewModels shared).

## Context

Table shares its UI between a MAUI Blazor Hybrid host and a Blazor Web App host via a Razor Class Library (ADR-0001). State management must work in both hosts — a pattern that runs natively in the MAUI `BlazorWebView` and in the Blazor Web App's interactive runtime (Server or, later, WebAssembly).

The options for Blazor state management:

1. **Scoped DI services** — plain services injected into components, holding state, calling `StateHasChanged`. Simplest, idiomatic Blazor, zero library.
2. **CommunityToolkit.Mvvm** — `ObservableObject` + `RelayCommand` + source-gen. The MAUI-standard MVVM toolkit; also works in Blazor. Lightweight, Microsoft-maintained.
3. **Fluxor** — Redux/Flux store, actions, reducers, devtools. Most structured; heaviest.

Forces at play:

- The app spans two hosts (MAUI + Blazor). A pattern that is idiomatic in *both* reduces friction; one that's Blazor-only fights the MAUI side and vice-versa.
- CommunityToolkit.Mvvm is the standard MVVM library for MAUI/WPF and is usable in Blazor — it is the natural bridge for a Hybrid app sharing ViewModels.
- Wallet uses Riverpod (Flutter's idiom); Table should use the .NET/MAUI idiom, not a direct Riverpod port.
- Lean stack preference (Board Games ADR-0003): avoid libraries that hide mechanics unless they pay off. MVVM via the toolkit is low-friction and transparent (source-gen, no magic runtime).

## Decision

Table uses **CommunityToolkit.Mvvm** for its ViewModels, shared across the MAUI and Blazor hosts from the RCL. ViewModels extend `ObservableObject` and expose `RelayCommand`/`AsyncRelayCommand` source-generated commands; components bind to them.

- **ViewModels co-located with their feature** in the RCL (`Pages/<screen>/<Feature>ViewModel.cs`), not in a global folder.
- **One ViewModel per screen/feature slice**, injected as scoped DI services; components bind to their properties/commands.
- The `ObservableObject` notification mechanism works in both the MAUI `BlazorWebView` and the Blazor Web App interactive runtime, so the same ViewModel drives both hosts without host-specific branching.

This is the lean, idiomatic-Hybrid choice: scoped DI services are simpler but less structured for an app that will grow screens (collection, catalog, accessories, sleeves, plays); Fluxor is over-structured for a collection manager. CommunityToolkit.Mvvm is the middle that also matches the MAUI idiom.

## Consequences

Positive:

- One ViewModel pattern works in both hosts — no Blazor-only vs MAUI-only state split.
- Source-generated commands (`[RelayCommand]`) reduce boilerplate; `ObservableObject` gives change notification without manual `INotifyPropertyChanged` plumbing.
- Co-located ViewModels keep each screen's state local and testable.
- Microsoft-maintained, mature, low-risk.

Negative:

- MVVM in Blazor is less idiomatic than in MAUI/WPF (Blazor's native idiom is "code-behind with services"). Using ViewModels in Blazor components is a slight departure from the most common Blazor style, though well-supported.
- One more library on the client. Justified by the cross-host unification.

Neutral:

- Scoped DI services remain in use for cross-cutting concerns (auth state, the `BoardGamesApiClient`, theme) — ViewModels are for per-screen feature state, not a replacement for all DI services.

## Alternatives Considered

1. **Plain scoped DI services (no library).** Viable and simplest; rejected as the *default* only because the Hybrid span (MAUI + Blazor) favours a pattern idiomatic to both, and CommunityToolkit.Mvvm is the MAUI standard that also works in Blazor. Scoped services remain for cross-cutting state regardless.
2. **Fluxor (Redux).** Rejected: the most structured option; overkill for a collection manager. The devtools/time-travel benefit doesn't justify the ceremony at this scale.
