# ADR-0004: Online-only Phase 1 — no offline store/outbox; Dapper for any future read cache

## Status

Accepted

Date: 2026-07-07

**Relates to:** [Wallet ADR-0004](../../../wallet/docs/adr/0004-offline-first-sync.md) (Wallet's offline-first model — the contrast Table explicitly does *not* adopt for Phase 1).

## Context

Wallet has a mature, ADR-frozen offline-first layer: a local drift cache of read models + a `pending_operations` write outbox + idempotent replay, with pending work shown-and-counted via a view-time overlay. It is the right design for a personal-finance *planner* the user opens constantly, including offline.

Table is a different kind of app — a board-game collection manager. The offline scenarios are real but narrower (checking the collection at a game cafe or store; logging a play offline) and not as critical as Wallet's financial planning. Re-implementing Wallet's outbox in Blazor (a local SQLite store + a background sync drainer + the overlay logic) is substantial work, and Table is already re-implementing auth/state/theme from scratch as part of the Blazor experiment (ADR-0001). Adding a full offline layer on top in Phase 1 bloats the first vertical without a proven need.

Forces at play:

- Table is online-only-capable by default (it calls the Board Games API directly); offline is an enhancement, not a requirement, for Phase 1.
- The host + guests scale is tiny (1 host, a handful of guests) — no multi-user conflict resolution needed even if offline lands later.
- Wallet's offline layer is ADR-frozen and proven; if Table ever needs the same, the pattern is documented and portable (with effort).
- A read cache (if needed later) is much simpler than a full write outbox — no replay/idempotency overlay, no failure classification.

## Decision

Table is **online-only in Phase 1**. Both the MAUI host and the Web host assume connectivity. There is no local SQLite store and no write outbox. Mutations go directly to the Board Games API via the typed `HttpClient`; reads fetch on demand.

**Future read cache (deferred):** if profiling reveals reads worth caching (e.g., the collection list fetched often, catalog detail rarely changing), a **read-only** local cache may be added. If so, it will be a lightweight **SQLite cache queried via Dapper** — not EF Core (avoids a second EF model on the client and keeps the client footprint small). No write outbox, no optimistic overlay, no sync drainer in Phase 1 — that complexity is deferred until a concrete offline-write need is proven, and would be its own ADR then.

This is a deliberate contrast with Wallet: Wallet is offline-first because its domain demands it; Table is online-first because its domain tolerates it, and the simpler client lets the Blazor experiment land faster.

## Consequences

Positive:

- Phase 1 client is dramatically simpler — no local DB, no outbox, no drainer, no overlay logic. The first vertical (auth → list → add) lands faster.
- No client-side reconciliation, failure classification, or sync-flicker surface (all real costs in Wallet).
- The Dapper-read-cache escape hatch is available later without committing to full offline writes.

Negative:

- The app is unusable without connectivity. Acceptable for Phase 1's scenarios (the host manages their collection at home / on the same network as the Gateway); a real limitation at a game cafe with no signal.
- A just-added item isn't visible if the network drops before the read refetches. Tolerable for a collection manager (not a planner where figures must always reconcile).

Neutral:

- The decision is reversible: if offline becomes important, a future ADR can add a read cache (Dapper) and, beyond that, a write outbox (porting Wallet's pattern). The online-only Phase 1 doesn't preclude either.

## Alternatives Considered

1. **Port Wallet's full offline-first layer to Blazor (drift → EF Core SQLite + outbox + overlay).** Rejected for Phase 1: substantial re-implementation work on top of an already-from-scratch auth/state/theme stack, for a domain whose offline need is narrower than a finance planner's. Deferred, not abandoned.
2. **EF Core SQLite for the future read cache (instead of Dapper).** Rejected: EF Core on the client is a second EF model to maintain, heavier than the cache use case needs. Dapper over SQLite is lighter and sufficient for a read-only cache. (The backend stays EF Core; the client uses Dapper only if a cache lands.)
3. **Online-only with no future cache plan.** Rejected as a permanent stance: a read cache is a likely later need, and naming Dapper now avoids a re-litigation. Online-only is the Phase 1 decision, not a forever decision.
