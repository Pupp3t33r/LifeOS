# ADR-0002: Catalog vs Collection separation

## Status

Accepted

Date: 2026-07-07

**Relates to:** [ADR-0001](./0001-full-ddd-and-domain-folder.md) (full DDD / bounded contexts), [ADR-0003](./0003-lean-stack.md) (lean stack incl. the external catalog abstraction).

## Context

Board Games pulls rich reference data from an external source (BoardGameGeek, or an alternative/fallback) — title, player count, play time, mechanics, themes, card composition, images. The same external game can be referenced by multiple hosts, and the external source corrects/enriches its data over time (image URL changes, a corrected player count, newly-published mechanics).

Separately, the host has a mutable relationship with each game: they wishlist it, own it, sell it, give it away. That ownership/lifecycle state is the host's own; it must never be disturbed by a catalog re-sync, and it carries fields the external source knows nothing about (acquisition date, disposition recipient, custom notes, condition).

There are two obvious ways to model this, and they behave very differently under sync and under multi-host use.

## Decision

Board Games keeps **two separate bounded contexts**: a read-mostly **Catalog** (externally-sourced reference data) and a mutable **Collection** (the host's ownership/lifecycle). They are distinct aggregate families in `Domain/` and distinct table sets, joined only by a foreign key from `CollectionItem.CatalogGameId` to `CatalogGame.Id`.

- **`CatalogGame`** (Catalog context) — keyed by `(Source, ExternalId)`, e.g. `("bgg", 13)`. Holds the immutable-ish reference data: title, year, image, player range, play-time range, complexity, rating, mechanics/themes (JSONB), card composition. Refreshed by the sync service; never carries ownership. Two hosts referencing *Catan* share one `CatalogGame` row.
- **`CollectionItem`** (Collection context) — the host's copy. Carries `Status` (Wishlist/Owned/Sold/GivenAway/Removed), acquisition/disposition fields, custom title/notes, condition, and an optional `CatalogGameId`. For ad-hoc accessories with no catalog entry (a generic chess timer), `CatalogGameId` is null and the item self-describes.

A re-sync updates `CatalogGame` rows in place; ownership state is untouched. A host deleting their `CollectionItem` does not delete the `CatalogGame`.

## Consequences

Positive:

- Re-syncing the catalog is safe and idempotent — it can never corrupt ownership state.
- Multiple hosts share catalog rows (one row per external game), avoiding duplicated reference data.
- Ownership fields that the external source doesn't model (disposition recipient, custom notes) have a natural home on `CollectionItem`.
- The two contexts can evolve independently — the sync cadence is decoupled from ownership writes.

Negative:

- Reads that need both (the collection list with catalog-enriched titles/images) require a join. Cheap at this scale (one host + guests); surfaced through read DTOs.
- An ad-hoc item (no catalog entry) carries its own descriptive fields, so there are two "title" sources (catalog title vs item custom title). Resolved by: display the custom title when set, else the catalog title.

Neutral:

- `CatalogGame` is an aggregate in its own bounded context, not a projection. It has its own repository and its own invariants (valid ranges, non-negative counts).

## Alternatives Considered

1. **Single denormalized `CollectionItem` that embeds the catalog fields.** Rejected: every re-sync would have to rewrite ownership rows (risk of disturbing state, and ambiguous — which copy wins if the host edited the title?). Duplicates reference data across hosts. Loses the "one catalog row per external game" sharing.
2. **Catalog as a projection over the external API (no local table).** Rejected: the external API (BGG) is rate-limited and unreliable; reads must not depend on it. A cached local `CatalogGame` table is the only sane read source, with sync populating it asynchronously.
3. **Catalog owned per-host (each host has their own copy of game data).** Rejected: wasteful duplication, and sync would have to fan out to every host's copy. The shared-row model is strictly better for reference data.
