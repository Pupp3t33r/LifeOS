# ADR-0006: Sleeve tracking model

## Status

Accepted

Date: 2026-07-07

## Relates to

[ADR-0002](./0002-catalog-vs-collection-separation.md) (card composition lives on `CatalogGame`, the reference side).

## Context

Sleeving board-game cards has three independent quantities that are often conflated:

1. **What sleeves the host owns** — a physical inventory: brand, size, color, count. "I have 200 standard matte sleeves from Brand X."
2. **How many cards of each size a game contains** — a property of the game itself (reference data): "Catan has 60 standard cards and 20 mini cards." This comes from the external catalog or manual entry.
3. **How many cards of each size are currently sleeved** for a specific copy of a game — the host's application of their inventory to that game: "I've sleeved 40 of Catan's 60 standard cards."

A common mistake is to treat sleeves as a consumable that *decrements* when applied (inventory goes down as you sleeve). In practice the host wants to know both: how many sleeves they own *and* how many cards of each game are sleeved (so they can see coverage — what's left to sleeve). Decrementing inventory on sleeving loses the "how many do I own" figure and makes "I removed the sleeves" ambiguous.

Forces at play:

- Card sizes are a finite vocabulary (standard, mini, tarot, mini-euro, etc.) but not perfectly standardized across publishers.
- A game's card composition is reference data (ADR-0002) — it belongs on `CatalogGame`, not on the host's copy.
- Sleeving status is per-host-copy: two hosts with the same game may have sleeved different counts.
- "Coverage" (cards needed − cards sleeved, per size, per game) is a derived read, not stored state.

## Decision

Three separate concepts, three separate stores:

1. **`SleeveBatch`** (Collection context) — the host's physical sleeve inventory. Fields: `Id`, `CollectionId`, `Brand`, `CardSize`, `Color`, `Count`, `Notes`. A batch is a box/batch the host owns; `Count` is how many they own. **Inventory is not consumed by sleeving** — applying sleeves to a game does not decrement a batch.
2. **`CardComposition`** (Catalog context, on `CatalogGame`) — a list of `{ CardSize, Count }` entries describing how many cards of each size the game contains. Reference data, synced from the catalog or entered manually. Lives on the catalog row so all hosts share it.
3. **`SleevingStatus`** (Collection context) — per `CollectionItem` (the host's copy of a game) + `CardSize`: how many cards of that size are sleeved. Constraint: `SleevedCount ≤ CardComposition.Count` for that size on the item's `CatalogGame`.

**Coverage** (cards unsleeved per size, per game) is computed at read time: `CardComposition.Count − SleevingStatus.SleevedCount`, per size. It is never stored as a snapshot.

## Consequences

Positive:

- The host always knows how many sleeves they own (`SleeveBatch.Count`) regardless of how they've been applied.
- "How many cards of game X still need sleeving?" is a pure read (coverage), always consistent with the latest composition and status.
- Card composition shared on `CatalogGame` (ADR-0002) means a catalog correction propagates to every host's coverage view automatically.
- Removing sleeves from a game is just lowering `SleevingStatus.SleevedCount` — no inventory "refund" logic.

Negative:

- `SleeveBatch.Count` can drift from reality if the host loses sleeves physically without recording it. That's a user-truth problem, not a model problem — the host edits the count.
- Cross-referencing three stores for the coverage view is a join; cheap at this scale, surfaced as a read DTO.

Neutral:

- Whether `SleeveBatch` is its own aggregate or part of `Collection` is deferred to implementation. The invariant (`SleevedCount ≤ composition count`) spans `SleevingStatus` and `CatalogGame`, so it's enforced in a domain service or the `Collection` aggregate at write time, not by a DB constraint alone.

## Alternatives Considered

1. **Sleeves as a consumable (inventory decrements on sleeving).** Rejected: loses "how many do I own" once applied; makes "I removed the sleeves" ambiguous (do they go back to inventory?); conflates ownership with application. The three-concept model keeps them cleanly separable.
2. **Store coverage as a snapshot column.** Rejected: a derived value that must stay consistent with composition + status; storing it invites drift. Compute at read.
3. **Card composition per-host-copy (on `CollectionItem`).** Rejected: it's reference data about the game, not the host's copy. Sharing it on `CatalogGame` (ADR-0002) means a catalog correction helps everyone and avoids duplication. A host can override via a custom field if they have a non-standard printing — deferred edge case.
