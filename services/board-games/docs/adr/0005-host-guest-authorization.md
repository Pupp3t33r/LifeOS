# ADR-0005: Host/guest authorization model

## Status

Accepted

Date: 2026-07-07

## Context

Board Games introduces a new authorization concept that Money/Wallet don't have: **guests**. Money is solo-only in v1 (admin/user roles only). Board Games has a host who fully controls a collection, and guests who get a read-only view (and, later, a limited write surface for rating/scoring).

Requirements gathered in planning:

- A collection has exactly one **host** — the user who owns and controls it.
- **Guests are not anonymous.** They must be authenticated LifeOS users (same `lifeos` realm, same SSO).
- "Anyone can be a guest for any host" — the guest role is not pre-provisioned per-user. Any authenticated user can become a guest of a collection, given the right access.
- Phase 1 is **host-only** (the app is the host's control surface). The guest UX is a later phase. But the data model must carry the host/guest distinction from day one so adding guests later is not a migration.

Forces at play:

- Guests share the same Keycloak realm as the host — they already have a `sub`. No separate identity store.
- "Anyone can be a guest" rules out a pre-provisioned guest list (no host-managed account creation, no per-guest invite table to maintain).
- Read-only guests must not be able to mutate the host's collection, even via the same API surface the host uses.
- The API enforces authorization; the app is host-only in Phase 1, so guest access arrives as a separate surface later.

## Decision

A `Collection` carries exactly one `HostUserId` (the controlling user's JWT `sub`) and a **`ShareToken`**. Authorization is **role-by-relationship-to-the-collection**, not a Keycloak role:

- **Host** — the user whose `sub == Collection.HostUserId`. Full control: CRUD on items, accessories, sleeves, plays; regenerate the `ShareToken`.
- **Guest** — any *other* authenticated LifeOS user presenting the collection's `ShareToken`. Read-only access to the collection's items (and, later, a limited write surface for rating/scoring once that feature lands).
- Everyone else — no access, even if authenticated.

The `ShareToken` is an opaque, regenerable secret stored on the `Collection`. A guest accesses via a share link/route that carries the token (e.g. `GET /shared/{shareToken}/collection`). The token's validity is checked against the stored value; regenerating the token revokes all outstanding guest access.

**Phase 1 (host-only):** the API enforces host-only writes on the main `/api/board-games/collection/*` surface (the caller's `sub` must equal `HostUserId`). The `ShareToken` exists on the model and can be generated, but the guest read endpoints (`/shared/{shareToken}/...`) and guest UX are deferred — the model is ready, the surface is not.

## Consequences

Positive:

- No per-guest provisioning — a host shares a link, any authenticated user becomes a guest. Minimal friction.
- Revocation is trivial (regenerate the `ShareToken`).
- Guests reuse the existing `lifeos` realm SSO — no new identity flow.
- The Phase 1 host-only app needs no authorization complexity; the model carries the guest seam forward without a migration.

Negative:

- A `ShareToken` is a bearer secret — if leaked, anyone authenticated can read until the host regenerates it. Acceptable for a personal collection; the token is regenerable and reads are non-destructive.
- "Anyone authenticated can be a guest" means there is no per-guest identity record until they actually access. A history of *who* viewed is not kept by default (no guest list). Acceptable for v1; a guest audit log is a later refinement if wanted.

Neutral:

- This is role-by-relationship, not a Keycloak role. The host is not a `host` role in Keycloak — they're the user whose `sub` matches `HostUserId`. A user can be a host of their own collection and a guest of someone else's, simultaneously, with the same token.

## Alternatives Considered

1. **Pre-provisioned guest list (host creates/invites guest accounts).** Rejected: more setup friction for the host, lower friction for the guest, and contradicts "anyone can be a guest." Requires an invite table and an account-creation/invite-accept flow. The ShareToken model is strictly simpler.
2. **Keycloak `guest` role + per-collection ACL.** Rejected: a realm-wide `guest` role doesn't capture "guest *of which collection*." Per-collection ACLs recreate the ShareToken model with more moving parts. Role-by-relationship-to-the-collection is the cleaner fit.
3. **Anonymous (unauthenticated) guest access via share link.** Rejected explicitly: guests must be authenticated (planning requirement). Anonymous access loses accountability and complicates the future rating/score write surface.
