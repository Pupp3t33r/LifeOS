# ADR-0004: Zero-trust JWT validation in Money

## Status

Accepted

Date: 2026-06-14

## Context

Root AGENTS.md §5 specifies the auth architecture for LifeOS:

- Keycloak realm `lifeos`, JWT bearer tokens.
- Gateway validates JWT signature and expiry.
- Gateway passes the `sub` claim downstream as `UserId`.
- Roles: `user` (own data), `admin` (manage family/household data).

The open question is whether Money **re-validates** the JWT itself, or trusts the Gateway and reads `UserId` from a header it sets.

Forces at play:

- Money is a financial service. Incorrect authn/authz has real consequences (data leaks across users, unauthorized transactions).
- Network misconfigurations can expose internal services, bypassing the Gateway.
- New debug endpoints or local dev setups may inadvertently allow direct access to Money.
- ASP.NET Core ships mature `JwtBearer` middleware that handles signature validation, JWKS refresh, and claim extraction with no custom crypto code.
- Industry norm for financial and regulated services is defense in depth — multiple independent layers of validation.

## Decision

Money will use **zero-trust JWT validation**. Money re-validates JWT signatures itself using ASP.NET Core's built-in `JwtBearer` middleware. Money fetches Keycloak's JWKS (public signing keys) at startup and refreshes them periodically per the framework's defaults.

The `sub` claim is read directly from the validated token's claims — Money does **not** accept an `X-User-Id` header from the Gateway. Authentication is enforced via `RequireAuthorization()` on endpoints. Data scoping (`owner_id == userId`) is enforced in handlers, with the option to extract into a Marten session wrapper later for defense-in-depth at the query layer.

The Gateway continues to validate JWT too — this is defense in depth, with each layer independent.

## Consequences

Positive:

- Money remains correct even if the Gateway is misconfigured, bypassed, or has a bug. Authentication does not depend on network isolation assumptions.
- Standard for financial services; eases future compliance conversations.
- Uses built-in ASP.NET Core middleware — no custom cryptographic code to maintain or audit.

Negative:

- Money requires network access to Keycloak for JWKS retrieval. This adds an infra dependency: Keycloak must be reachable from Money at startup (Aspire handles this in dev; production deployment must preserve connectivity).
- Adds ~microseconds per request for signature validation. Negligible at expected volume.
- One more moving part in the Aspire AppHost topology (Money now depends on Keycloak starting up).

Neutral:

- The Gateway still validates JWT as the first layer of defense — Money's validation is a second, independent layer.
- JWKS refresh is automatic; no operational burden beyond keeping Keycloak reachable.

## Alternatives Considered

1. **Perimeter trust.** Money trusts the Gateway, reads `UserId` from an `X-User-Id` header set by Gateway after it validates the JWT. Rejected: inappropriate for a financial service. Depends on network isolation that may not hold in dev, debug, or future deployment topologies.
2. **Custom auth middleware.** Hand-rolled JWT validation. Rejected: ASP.NET Core's `JwtBearer` middleware handles every needed concern (signature, lifetime, audience, issuer, JWKS refresh) with mature, audited code. No benefit to custom implementation.
