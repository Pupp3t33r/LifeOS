# ADR-0003: Idempotency via client-assigned UUIDs

## Status

Accepted

Date: 2026-06-14

## Context

Money is a financial service. Duplicate transactions, recurring payments, or wishlist items cause real harm — incorrect balances, doubled rent payments, miscounted budgets. The threat model includes several scenarios that produce duplicate requests:

- User double-taps a Save button before the UI disables it.
- Mobile network drops mid-response (server processed, client never saw 200) — client must retry to know.
- Future offline-first mobile sync engines replay queued requests minutes, hours, or days later.
- Browser refresh during a pending fetch re-fires the request.

Root AGENTS.md §4.5 mandates that all mutation endpoints handle duplicate requests safely via "idempotency keys or natural keys."

The simplest mechanism that protects against these threats without speculative infrastructure is preferred.

## Decision

Money will use **client-assigned UUIDs as primary keys** for all creational endpoints. The client generates a UUID at form/draft instantiation (not on submit click) and sends it as part of the request. Money uses this UUID as the entity's ID.

For event-sourced aggregates, dedup is enforced as an **aggregate invariant**: the stream tracks recorded IDs and refuses to record the same ID twice. The handler returns the existing entity (200 OK) when a duplicate request arrives with matching payload, or signals 409 Conflict when the same ID arrives with a different payload (indicating a client bug).

Non-creational mutations (updates, deletes, state transitions) do not use idempotency keys. They use **optimistic concurrency** (ETag / `If-Match`) to prevent concurrent modification races — a separate concern.

## Consequences

Positive:

- Zero additional infrastructure. No `IdempotencyRecord` table, no TTL, no cleanup job.
- The aggregate enforces dedup naturally as part of its invariants — exactly what aggregates are for.
- Enables optimistic UI: clients can render entities locally before the server responds, knowing the server will deduplicate any replay.
- Works transparently with offline-first sync engines, which replay requests as originally sent (including the client-generated ID).
- Aligns with industry practice (Stripe, Square, and most modern financial APIs use the same pattern).

Negative:

- Only protects creational endpoints. Non-creational mutations need a separate concurrency story (ETags).
- Client must generate UUIDs correctly — tied to form instance, not click — or dedup fails silently. This is a documented client contract.
- Same ID arriving with different payload is treated as a 409 Conflict, which surfaces client bugs but requires clients to handle the response.

Neutral:

- This pattern is independent of the data layer; it works identically in Marten (event-sourced aggregates) and EF Core (other services).

## Alternatives Considered

1. **Server-side `Idempotency-Key` store.** Clients send an `Idempotency-Key` header; server stores `(key, request_hash, response)` for a TTL. Rejected for now: speculative infrastructure — there is no offline-first mobile consumer yet. The client-assigned UUID pattern provides equivalent protection with zero infra. Revisit if non-creational endpoints demand server-side idempotency.
2. **Client-side-only prevention (debounce, single-fire buttons).** Rejected as sufficient: client logic cannot prevent retries caused by network failures, browser refresh, or sync engine replays. Useful as a first line of defense but not a complete solution.
