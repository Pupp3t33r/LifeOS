# ADR-0012: Production schema migration policy (no runtime auto-create)

## Status

Accepted

Date: 2026-06-16

## Context

Money uses Marten (document + event store) and Wolverine (durability/outbox tables in a dedicated `wolverine` schema). Both can create their own PostgreSQL schema objects at runtime via Marten's `AutoCreate` setting. Until now the service ran `AutoCreate.All` in every environment, meaning the application mutated the database schema on startup and on first use of each document/projection type.

Forces at play:

- Runtime schema mutation requires the application's database role to hold DDL privileges in production — a standing risk (a compromised or buggy app can alter or drop schema).
- Schema changes applied implicitly at startup are invisible to review and to the deployment pipeline; drift between code and database is detected only by failure.
- The Critter Stack ships first-class tooling (the `JasperFx` command line: `resources setup/check`, `db-apply/db-assert/db-patch/db-dump`, `codegen write`) designed for exactly this — applying schema and pre-generating code as explicit, out-of-band steps.
- There is no CI/CD pipeline or deployed production environment yet; the *policy* can be fixed now while the *machinery* is built later.
- Local development and integration tests benefit from auto-create against throwaway databases (Aspire Postgres, Testcontainers) and should not pay the migration-pipeline cost.

## Decision

We will **never mutate the production database schema at application runtime.** Marten `AutoCreateSchemaObjects` is environment-gated: `AutoCreate.All` in Development, `AutoCreate.None` in every other environment. A non-Development app expects its schema — Marten document/event tables, Marten projection tables, and Wolverine durability/outbox tables — to already exist, and fails fast if it does not.

Schema migration in deployed environments is a **dedicated, privileged pre-deploy step**, separate from running the app: `dotnet run -- resources setup` (covering all stateful resources in one pass) executed under a DDL-privileged database role, with the application itself running under a least-privilege (DML-only) role. The pipeline gates on `dotnet run -- resources check` to fail on drift before deploy, and pre-generates handler code with `dotnet run -- codegen write` + `TypeLoadMode.Static` so production performs no runtime code generation. The operational wiring (enabling `RunJasperFxCommands`, the migration job/init-container, role separation, the codegen build step) is tracked in [PLAN.md §8](../../PLAN.md) and lands with CI/CD; this ADR freezes the *policy*, not the rollout date.

## Consequences

Positive:

- The production app role no longer needs DDL privileges — smaller blast radius.
- Schema changes become explicit, reviewable pipeline artifacts; drift is caught by a CI gate, not a production incident.
- One command (`resources setup`) covers Marten and Wolverine schema together, avoiding partial migrations.
- Local/dev/test iteration is unchanged (auto-create against disposable databases).

Negative:

- Deploys gain a mandatory ordering: migrate before rolling out app instances. A missed migration step makes the app fail fast on startup (by design, but a new failure mode to operate).
- Requires building the migration job, role separation, and codegen step before the first real production deploy — work that does not exist yet.
- `TypeLoadMode.Static` requires generating/committing handler code as part of the build; skipping it degrades to runtime generation or a startup failure.

Neutral:

- Integration tests run as Development explicitly so they keep auto-creating; this couples test correctness to the environment name, documented at the call site in `MoneyApiFactory`.

## Alternatives Considered

1. **Keep `AutoCreate.All` everywhere.** Rejected: requires DDL privileges in production and applies unreviewed schema changes at runtime — both unacceptable for a deployed environment.
2. **`AutoCreate.CreateOrUpdate` in production** (additive only, never drops). Rejected: still requires runtime DDL privileges and still applies changes implicitly; safer than `All` but does not meet the "no runtime mutation" bar.
3. **Hand-written SQL migrations via Flyway/DbUp as the primary mechanism.** Deferred, not rejected: `db-patch`/`db-dump` can feed reviewable DDL into such a tool, and stricter environments may adopt it. For now tool-applied `resources setup` is sufficient and lower-maintenance.
