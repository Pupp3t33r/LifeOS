# ADR-0002: Event versioning strategy for Money

## Status

Accepted

Date: 2026-06-14

## Context

Money uses Marten event sourcing on PostgreSQL. Events appended to streams are immutable historical facts — once written, they cannot be modified or deleted without explicit, dangerous rewrite operations. As the domain model evolves, the C# event types will need to change: fields get added, semantics shift, mistakes get corrected.

The question is how to handle the gap between old events already in storage and new code that must read them.

Money is currently pre-release. No data exists in production; the database can be dropped and recreated freely. After the first release, historical events must remain readable forever.

Forces at play:

- Marten supports multiple versioning strategies (upcasters, multiple event types, additive-only evolution).
- Solo dev, no QA team — strategies that require careful testing of transformation code carry more risk.
- Money AGENTS.md mandates Marten as the only data access tool.
- Once consumers exist for Money's Kafka events (Planner, etc.), event shape changes ripple to them as well.

## Decision

We will use a **two-phase event versioning policy**:

**Dev mode (pre-v1.0):** The Marten database may be dropped and recreated freely. Event types may be modified in place — including renaming fields, removing fields, and changing semantics. No backward compatibility is required because no real data exists. This maximizes development velocity.

**Post-release (v1.0+):** Breaking changes are handled by **introducing new event types** alongside the original (e.g., `TransactionRecordedV2`). Old event types remain unchanged in code and storage. Aggregates register `Apply` methods for every version they may encounter. New code writes only the newest version; old events in storage deserialize as their original type and are routed to the matching `Apply` method.

Notably, this policy also forbids **nullable additions** to existing event types post-release. Even purely additive changes are expressed as new event types. This is stricter than the typical "additive-only" pattern but eliminates ambiguity for consumers and aggregate authors.

## Consequences

Positive:

- Dev velocity is unconstrained until first release.
- Post-release, no risk of historical events becoming unreadable — old types are immutable in storage and code.
- Aggregate code is explicit about which versions it handles; no magic transformations.
- Consumers (Kafka) can subscribe to specific event versions; version transitions are deliberate.

Negative:

- Post-release, aggregates accumulate `Apply` methods for each historical event version. Over years this can become cluttered.
- Retiring legacy event types requires verifying no live streams still produce them — non-trivial without tooling.
- New event types must be propagated to all consumers explicitly (Kafka topics, consumer deserializers).

Neutral:

- Stricter than typical "additive-only with nullable fields" — chosen for clarity over minimalism. The cost is more event types; the benefit is no nullable-field handling in consumers or aggregates.

## Alternatives Considered

1. **Additive-only evolution with nullable fields.** Add new optional fields to existing event types; old events deserialize with nulls for missing fields. Rejected post-release: nullable handling complicates consumer logic, and silent nulls can mask real bugs in a financial domain.
2. **Marten upcasters (transform-on-read).** Keep a single event type name; write transform code that converts old storage shape to new in-memory shape at read time. Rejected: upcasters run forever once introduced, and a bug in an upcaster causes silent data corruption. Too risky for a solo dev without dedicated QA.
