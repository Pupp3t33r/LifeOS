# ADR-0001: REST contract approach for Money

## Status

Accepted

Date: 2026-06-14

## Context

Money is the first service being built in LifeOS. Before any endpoint code is written, we need to decide how the REST API contract is authored, exposed, and consumed.

Forces at play:

- Solo developer, learning project — velocity and minimal tooling overhead matter.
- .NET 10 ships built-in OpenAPI support via `Microsoft.AspNetCore.OpenApi`, making code-first generation essentially free.
- No frontend (web or mobile) exists yet — there is no immediate consumer demanding a frozen, parallelizable contract.
- Money AGENTS.md "Feature Organization" mandates Minimal APIs in vertical-slice feature folders (`Features/Transactions/RecordExpense.cs`).
- Root AGENTS.md §4.10 mandates Minimal APIs over controllers.

The choice of contract approach constrains how frontend, mobile, and any future third-party consumers integrate with Money.

## Decision

We will use **code-first REST** with Minimal APIs in vertical-slice feature folders. Endpoint definitions and request/response types in C# are the single source of truth for the contract. We will use `Microsoft.AspNetCore.OpenApi` (built into .NET 10) to generate the OpenAPI spec at runtime, served at `/openapi/v1.json` with Swagger UI at `/swagger`.

The published spec is documentation derived from code — not a frozen, versioned artifact that consumers code against.

## Consequences

Positive:

- Fastest path to working endpoints; no codegen toolchain to maintain.
- Single source of truth — code and spec cannot drift.
- Aligns with .NET idioms and existing AGENTS.md conventions (Minimal APIs, vertical slices).
- Spec auto-updates as code changes; zero manual sync work.

Negative:

- Frontend and mobile cannot freeze a contract to code against until endpoints exist.
- The published spec is not formally versioned; downstream consumers must accept that it reflects current code.
- Endpoint descriptions and examples require XML comments or attributes to surface in the spec — discipline required to keep the spec useful.

Neutral:

- Nothing in this choice prevents switching to spec-first later. The generated spec can be extracted, frozen, and made authoritative if parallel frontend work demands it.

## Alternatives Considered

1. **Spec-first OpenAPI with codegen (Kiota / NSwag).** Hand-write OpenAPI YAML, generate C# types and endpoint stubs. Rejected: tooling overhead for a solo dev; introduces two sources of truth (spec and code) that must be reconciled; no immediate consumer justifies the cost.
2. **Hybrid (code-first REST, spec-first events).** Code-first for endpoints, hand-written AsyncAPI for Kafka events. Deferred: a reasonable pattern, but Money has no Kafka events in flight yet. Revisit when the first cross-service event is wired.
